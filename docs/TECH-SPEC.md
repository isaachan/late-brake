# Late Brake - Technical Specification v1

## 1. 整体软件架构

### 1.1 架构设计原则

- **模块化分层**：各模块职责单一，依赖清晰
- **纯Python实现**：无系统级依赖，易于安装和分发
- **CLI 优先**：遵循 UNIX 哲学，单次命令完成单次任务，支持脚本化
- **数据驱动**：统一内部数据格式，各模块通过标准结构交互
- **可扩展**：输入格式、输出格式易于扩展，不影响现有代码

## 2. 关键技术与算法

### 2.1 GPS 距离计算

**技术选型**：使用 [GeographicLib](https://geographiclib.sourceforge.io/) 计算球面距离和方位角。
- 相比于 Haversine 公式，GeographicLib 精度更高，处理椭球面模型更准确
- 对于赛车场尺度（几公里），误差可以忽略不计
- Python 绑定 `geographiclib` 纯 Python 实现，无需编译

**用途**：
- 累计行驶距离计算
- 点到线距离计算（起终点线检测）
- 轨迹点到赛道中心线投影

### 2.2 自动圈速分割算法

**核心思路**：基于起点/终点门（gate）线穿越检测：
1. 预处理：对原始数据按时间排序，计算每个点累计距离
2. 穿越检测：检测轨迹点何时从 gate 线一侧穿越到另一侧
3. 方向验证：仅保留行驶方向正确的穿越（避免掉头/反方向行驶误判）
4. 圈分割：两次正确穿越之间的数据点切分为一个单圈
5. 完整性判断：首圈末圈如果未完整穿越，标记为不完整

**关键参数**：
- 穿越检测阈值：点到 gate 线的最小距离（默认 5 米）
- 最小圈时差：避免连续误穿越（默认 30 秒，可配置）

### 2.3 圈速对比对齐算法

两个圈对比时，需要按距离对齐采样点（因为两个圈的走线长度不同，采样频率也可能不同）：

**算法步骤**：
1. 将两个圈都按累计距离重采样（默认 1 米间隔）
2. 对每个距离点，插值计算对应速度、G值等参数
3. 按分段/弯道分段计算时间差和平均速度差
4. 输出整体对比结果和分段对比结果

**使用技术**：`numpy.interp` 线性插值，足够精确且性能良好。

### 2.4 赛道自动匹配算法

基于 GPS 轨迹和赛道锚点（anchor）匹配：

1. 计算输入数据的中心点经纬度
2. 计算中心点与所有已配置赛道锚点的距离
3. 如果最小距离小于锚点半径，则匹配成功
4. 如果多个赛道都匹配成功，返回歧义错误，要求用户手动指定

### 2.5 数据平滑（可选，v1 可不实现）

GPS 采样存在噪声，可使用滑动平均或 Savitzky-Golay 滤波对数据进行平滑。

- 依赖：`scipy`（如果需要此功能，可作为可选依赖）
- v1 暂不强制要求，后续迭代添加

## 3. 技术选型

### 3.1 语言与核心依赖

| 组件 | 选型 | 理由 |
|------|------|------|
| 语言 | Python 3.10+ | 纯 Python 易于开发分发，数据处理生态丰富 |
| CLI 框架 | Click | 成熟稳定，生态好，支持子命令，易于扩展 |
| 数据验证 | Pydantic v2 | 类型安全，运行时验证，序列化方便 |
| 地理计算 | GeographicLib | 精度高，纯 Python，无依赖 |
| 数值计算 | NumPy | 插值、重采样、统计计算标准库 |
| JSON Schema | jsonschema | 赛道 JSON 文件验证 |
| 测试 | pytest | Python 标准测试框架 |

### 3.2 为什么不选择其他语言？

- **Go/Rust**：编译型，分发需要跨平台编译，对于个人工具项目，Python 开发效率更高
- **JavaScript/TypeScript**：CLI 体验不如 Python，地理计算库生态弱
- **C++**：开发周期长，对于这个量级的工具过度工程化

结论：Python 是最佳平衡点，满足需求，开发效率高，易于维护。

## 4. 核心约束条件

1. **纯 CLI**：无 Web UI，无 GUI，所有功能通过命令行完成
2. **单次执行**：每个命令执行完成后直接退出，不保留进程（无 REPL）
3. **Python 生态**：必须基于 Python 生态，不能引入非 Python 依赖
4. **数据隐私**：所有数据处理在本地完成，不上传云端，保护用户隐私
5. **单位统一**：所有内部计算采用公制单位（米，秒，km/h，度）

## 5. 风险分析与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| GPS 采样噪声大，分割不准 | 圈速分割错误 | 1. 增加距离滤波；2. 允许用户后续手动调整；3. 输出分割结果供检查 |
| 不同记录仪时间/距离基准不一致 | 对比结果不准 | 统一内部重新计算累计距离，不依赖数据源输出的距离 |
| 赛道匹配错误 | 整个分析错误 | 1. 锚点半径合理设置；2. 歧义时明确提示要求用户指定；3. 支持 `--track` 强制指定 |
| NMEA/VBO 格式变体多 | 解析失败 | 1. 对已知格式变体做兼容；2. 解析错误给出明确错误信息；3. 支持 verbose 输出调试 |
| 大文件内存占用高 | 性能下降 | 数据文件通常 < 10MB，Python 完全可以处理，v1 无需流式处理 |

## 6. 核心数据模型

基于 Pydantic 定义，详见 [data-format.md](./data-format.md) 和 [track-format.md](./track-format.md)。

核心模型对应关系：

- `DataPoint` → 单个采样点
- `Lap` → 单圈数据
- `Track` → 赛道元数据
- `TrackSector` → 赛道分段
- `TrackTurn` → 弯道信息

## 7. 模块职责定义

### 7.1 CLI 层 (`late_brake.cli`)

- 使用 Click 定义根命令和子命令
  - `track list` / `track info` / `track add`
  - `load`
  - `compare`
- 处理命令行参数
- 调用核心逻辑
- 按格式输出结果（文本美化或 JSON 序列化）

### 7.2 核心层 (`late_brake.core`)

- `splitter`：根据赛道 gate 线分割单圈
- `comparator`：按距离重采样对齐，计算对比结果
- `matcher`：根据 GPS 轨迹自动匹配赛道

### 7.3 模型层 (`late_brake.models`)

- Pydantic 模型定义
- 数据验证逻辑
- 类型提示

### 7.4 IO 层 (`late_brake.io`)

- `parsers`：各种输入格式解析器，输出统一 `List[DataPoint]`
- `track_store`：管理内置和用户自定义赛道，读取、添加、列出
- `exporter`：输出结果序列化为文本或 JSON

## 8. 后续技术演进方向

- **可视化**：输出可直接导入 Matplotlib/Excel 的曲线数据
- **刹车点分析**：专门提取刹车点（纵向 G 值由正变负，速度开始下降的位置）对比
- **最优走线计算**：基于多圈数据找出每个点最快速度对应的走线
- **滤波平滑**：添加可选的数据平滑算法，减少 GPS 噪声影响
- **更多格式**：支持 CSV, AiM 等更多输入格式
